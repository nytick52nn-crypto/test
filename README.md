# Тестовое задание МАВИКО — Python-разработчик на автоматизацию

Три задачи, четыре модуля в `src/`:

- `ozon_client.py` — общий клиент Ozon Seller API (ретраи, бэкофф на 429/5xx, пагинация),
  используется задачами 1 и 2.
- `export_products.py` — задача 1: выгрузка товаров в CSV.
- `telegram_summary.py` — задача 2: утренняя сводка в Telegram на основе выгрузки.
- `clean_catalog.py` — задача 3: чистка каталога (без API, работает только с CSV).

## Установка и запуск

Создайте `.env` в корне проекта со следующими переменными:

```
OZON_CLIENT_ID=...
OZON_API_KEY=...
OZON_BASE_URL=https://api-seller.ozon.ru   # можно не указывать — это значение по умолчанию

TELEGRAM_BOT_TOKEN=...     # токен от @BotFather
TELEGRAM_CHAT_ID=...       # см. https://api.telegram.org/bot<TOKEN>/getUpdates

LOW_STOCK_THRESHOLD=5      # порог низкого остатка для пометки «заканчивается»
```

```bash
pip install -r requirements.txt
cd src

python export_products.py                       # -> data/products_export.csv
python telegram_summary.py --dry-run             # проверить текст сводки без отправки
python telegram_summary.py                       # отправить в Telegram
python clean_catalog.py --in ../data/catalog_raw.csv --out ../data/catalog_clean.csv
```

## Тесты

```bash
pip install -r requirements.txt pytest
pytest tests/ -v
```

48 тестов, все проходят: чистая логика `clean_catalog.py` (парсинг цены/бренда/OEM/упаковки,
дедуп, полный прогон на реальном `data/catalog_raw.csv`), retry/backoff-логика `ozon_client.py`
(429/5xx/сетевые ошибки — через мок HTTP-ответов, без реальных запросов) и чистые функции
`export_products.py`/`telegram_summary.py` (сборка сводки, разбиение на сообщения, разбор ответа API).

Обе задачи, завязанные на внешние сервисы, дополнительно проверены вживую, а не только
на тестах и моках:
- Задача 1 — реальный вызов тестового кабинета Ozon из задания (`python export_products.py`):
  `/v3/product/list`, `/v5/product/info/prices`, `/v4/product/info/stocks` реально отвечают
  и отдают корректные данные.
- Задача 2 — реальный Telegram-бот (создан через BotFather) и реальная отправка сообщения
  через `python telegram_summary.py` (без `--dry-run`) в личный чат — сводка с алертом по
  низкому остатку пришла и отобразилась корректно, включая эмодзи.

Ключи (Client-Id, Api-Key, токен бота, chat_id, порог остатка) — только в `.env`,
в коде не хардкодятся; `.env` в `.gitignore`.

## Как работали с документацией и ИИ

С Ozon Seller API до этого проекта не работал — специализация последние годы
BI/аналитика и автоматизация отчётности (Python, ETL, Airflow), а не интеграции
с маркетплейсами. Поэтому процесс выстроил в два шага, а не «сразу код»:
сначала — актуальность методов и требований по документации, и только потом —
реализация. Модели (Claude) ставил задачу не «напиши скрипт», а сначала —
свериться с документацией и changelog и определить, какие эндпоинты сейчас
действующие для трёх операций (список товаров, цены, остатки), а какие уже
устарели.

Конкретно передавал модели:
1. Требования к каждой задаче своими словами (что нужно выгрузить, как
   обрабатывать ошибки и лимиты, чем батч отличается от одиночного запроса) —
   отдельно по задачам 1, 2 и 3, а не всё сразу, чтобы не размывать контекст
   и было проще проверять ответ на конкретный вопрос.
2. Прямые вопросы вида «какой метод сейчас актуален для получения списка
   товаров продавца — /v2/product/list или /v3/product/list, и когда старый
   отключили».
3. Просьбу свериться с changelog-каналом Ozon Seller API (Telegram-канал
   с историей изменений методов), а не только с общей документацией — потому
   что именно там фиксируются даты отключения старых версий методов.
4. Просьбу построчно объяснить логику после реализации — чтобы проверить, что
   предложенные решения (например, дедуп по sku, а не по названию товара)
   обоснованы, а не произвольны, и суметь их аргументировать.

## Как проверяли, что модель не выдумала методы и не использовала устаревшие

- Каждый метод (`/v3/product/list`, `/v3/product/info/list`,
  `/v5/product/info/prices`, `/v4/product/info/stocks`) сверялся минимум по
  двум независимым источникам: официальный changelog Ozon Seller API
  (объявления вида «/v2/product/list устарел, отключается такого-то числа,
  используйте /v3/product/list») и сторонние технические статьи/примеры кода
  (например, интеграционные библиотеки), где показан реальный JSON запроса/ответа.
- Отдельно проверяли, что версия метода не более старая, чем нужно: например,
  для `/v1/product/info/prices` и `/v4/product/info/prices` в changelog прямо
  написано, что их отключили в пользу `/v5/product/info/prices` — если бы
  модель предложила `/v1` или `/v4`, это бы всплыло на этом шаге.
- Формат курсорной пагинации (`last_id` / `cursor`, а не `offset`) проверяли
  по примеру реального ответа API (`last_id` в `result`), а не полагались на
  память модели о формате — старые версии документации ещё используют `page`/
  `offset`, и не глядя в актуальный пример легко было бы воспроизвести
  устаревшую схему.
- Финальная проверка — не чтение документации, а факт: код покрыт 48 юнит-тестами
  и прогнан вживую против тестового кабинета Ozon (детали и результат — в разделе
  «Тесты» выше). Реальный ответ `/v3/product/list` с курсором `last_id` в `result`
  подтвердил формат пагинации окончательно — точнее прямого ответа API источника нет.

## По задаче 3: какие крайние случаи заметили и где данные были неоднозначными

1. **Список брендов надо строить по фактическим данным, а не наугад.** Первая
   версия `BRAND_HINTS` содержала обобщённые бренды автозапчастей (Bosch, TRW,
   FEBI...) — ни один из них не встретился в реальном каталоге ни разу. В
   `catalog_raw.csv` встречаются MAVICO (почти все строки), DBA и Деталиус —
   их и добавили в список. Урок: список брендов должен собираться по реальному
   каталогу продавца, а не подбираться по общим знаниям об отрасли.
2. **Дедупликация по sku — правильный выбор именно для маркетплейс-каталога.**
   По названию+цене дедуплицировать рискованно: два разных товара с одинаковым
   названием и случайно совпавшей ценой ложно схлопнутся. `offer_id` в реальных
   данных почти всегда есть и однозначно определяет товар, а один и тот же
   товар выгружается по нескольку раз с разным написанием названия (например,
   `MV1028F` встречается трижды: «Колодки тормозные передние Mavico MV1028F
   Lada Vesta комплект 4 шт», «колодки тормозные перед. mavico mv1028f lada
   vesta (компл 4шт)», «Колодки Mavico MV1028F Vesta 4 шт.» — с разным
   регистром и сокращениями). Поэтому `clean_catalog.py` дедуплицирует по
   `offer_id`, когда он есть и непуст, и откатывается на (название, цена)
   только для строк без sku (в файле — одна строка без цены и без offer_id:
   «Диск тормозной передний Mavico Solaris (без цены, уточнить)»). При
   совпадении sku, но разной цене между дублями, в лог пишется предупреждение.
3. **Собственный артикул товара дублируется в названии.** Название почти всегда
   содержит тот же код, что и в `offer_id` (`MV1028F`, `MF-1041`, `AF2205`...),
   написанный то слитно, то через дефис, то через пробел. Это не OEM-номер
   (OEM — это код детали *другого* производителя, на которую данная деталь
   является заменой/аналогом, например `OEM 8200123456` для Renault Logan или
   `OEM 2101-3502090` для Lada) — но раз этот код уже есть в колонке `sku`,
   оставлять его же в `clean_name` избыточно. Скрипт вырезает его из названия
   отдельным шагом, сопоставляя все варианты написания через regex по sku.
   Важная оговорка: это применяется только если sku буквенно-цифровой — для
   голого числового id вырезание отключено, потому что оно почти наверняка
   случайно совпадёт с числом штук в названии.
4. **Упаковка/количество описываются гораздо разнообразнее, чем предполагалось.**
   Кроме «N шт» и «комплект» в реальных данных встретились сокращение «к-т»
   («OEM 8200123456 к-т 2шт»), «набор» («набор 3 шт») и «пара» без числа вообще
   (подразумевает 2 шт). Все четыре формы добавлены как синонимы «комплекта».
   Отдельная ловушка: «40х40» (размер полотенца) — цифра перед «х» ошибочно
   принималась за число упаковки в первой версии регулярки; добавлена проверка,
   что число-перед-ключевым-словом не может само быть частью пары «ЦИФРАxЦИФРА».
5. **Цена без единиц измерения** (например, «250» без «руб»/«р») — считается
   рублями по умолчанию; это предположение, а не факт, отдельной валюты в
   реальном файле не встретилось, но если появится — автоматически не
   отличить.
6. **Что считать «пустой строкой».** Строка с пустым названием и пустой ценой
   (в файле — одна полностью пустая строка) отбрасывается; строка с названием,
   но без цены (тот же товар «Mavico Solaris (без цены, уточнить)») — оставляется
   с пустым `price_rub`, чтобы её было легко найти и заполнить вручную, а не
   потерять товар целиком.
7. **Остаток (`stock`) сохраняется отдельной колонкой** в `catalog_clean.csv` —
   в задании это явно не требовалось, но раз колонка есть в сыром файле, отбрасывать
   её молча показалось неправильным (при дедупе по sku остаток берётся из первой
   оставленной записи).

## Требования из задания и как они закрыты

- Ключи из `.env`, не в коде — везде через `os.environ` / `python-dotenv`.
- Обработка HTTP 429 и лимитов с повторными попытками — `OzonClient._post`:
  экспоненциальный бэкофф, уважает `Retry-After`, до 5 попыток по умолчанию.
- Запросы батчами — цены и остатки запрашиваются пачками по 100 `product_id`
  за вызов (настраивается параметром `batch_size`).
- Логирование — во всех трёх скриптах, в консоль и в `logs/*.log`.
- Консоль Windows (частая кодировка cp1251) не роняет вывод: во всех трёх
  скриптах stdout/stderr явно переключаются на utf-8, иначе эмодзи в сводке
  Telegram (📦/⚠️) вызывали `UnicodeEncodeError`, а кириллица в логах
  отображалась кракозябрами.
